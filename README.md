# PLDF - Progressive Learning Development Framework

**PLDF (Progressive Learning Development Framework)** - это методичный образовательный фреймворк для обучения студентов разработке программного обеспечения с помощью LLM-агентов. В отличие от других фреймворков, PLDF фокусируется не только на технической реализации, но и на образовательном процессе, помогая студентам понимать **ПОЧЕМУ** принимаются те или иные решения.

### Особенности

- ✅ Начинает с концепции проекта, а не с технических ограничений
- ✅ Дизайн UI перед техническими решениями (исключает ненужные эндпоинты)
- ✅ Обоснованный выбор технологий с обсуждением
- ✅ Пошаговая реализация с автоматической валидацией
- ✅ Отслеживание прогресса (можно возобновить с любого шага)
- ✅ Проверка понимания на каждом этапе (мини-тесты и практические задания)
- ✅ Review и рефлексия для закрепления знаний

## Философия

PLDF построен на принципе **методичного подхода** к разработке:

1. **Начинаем с концепции**, а не с технических ограничений
2. **Дизайн UI перед техническими решениями** - создание интерактивных HTML/CSS прототипов, которые можно открыть в браузере
3. **Обоснованный выбор технологий** - студент понимает, почему выбирается та или иная технология
4. **Пошаговая реализация** с автоматической валидацией
5. **Отслеживание прогресса** - можно возобновить работу с любого шага

## Структура этапов

### Этап 0: Концепция проекта (`/pldf.concept`)
Определение проблемы, целевой аудитории, основных функций и критериев успеха.

**Особенности**:
- Автоматическая инициализация GIT репозитория с детальным `.gitignore` (исключает секреты, зависимости, временные файлы)
- Автоматическая валидация концепции перед переходом к следующему этапу
- Запись решений в Decision Log (`.pldf/project-context.md`)
- Первый коммит в GIT после создания концепции

**Выходные артефакты**:
- `.gitignore` - файл исключений для GIT репозитория
- `.pldf/concept.md` - полная концепция проекта
- `.pldf/project-context.md` - контекст проекта с Decision Log

### Этап 1: Дизайн UI (`/pldf.design`)
Создание интерактивных HTML/CSS прототипов интерфейса на основе концепции. Прототипы можно открыть в браузере (включая встроенный браузер Cursor) для визуального просмотра и утверждения.

**Особенности**:
- Прототипы создаются как реальные HTML/CSS файлы, которые можно открыть в браузере
- Прототипы создаются с учетом платформ из концепции - мобильные паттерны для Mobile, десктопные для Desktop, адаптивные для Web
- Создание локального сервера для просмотра прототипов (Go, Python или Node.js)
- Автоматическая валидация дизайна (покрытие функций, логика потоков, соответствие платформам)
- Запись решений о паттернах UI и структуре навигации в Decision Log
- **Требование утверждения дизайна студентом перед продолжением** - агент не перейдет к следующему этапу без утверждения

**Выходные артефакты**:
- `.pldf/ui-prototype/` - директория с HTML/CSS прототипами
  - `index.html` - главный файл с навигацией
  - `[screen-name].html` - HTML файлы для каждого экрана
  - `README.md` - инструкция по запуску локального сервера
  - Локальный сервер (если создан)
- `.pldf/ui-design.md` - описание дизайна с ссылками на прототипы
- `.pldf/project-context.md` - обновленный контекст проекта

### Этап 2: Выбор технологий (`/pldf.tech`)
Выбор технологического стека с обоснованием и обсуждением со студентом.

**Особенности**:
- Предложение 2-3 вариантов для каждого компонента стека с детальным обоснованием
- Обсуждение каждого выбора со студентом (не переход к следующему этапу без согласия)
- Консультация по структуре проекта (Feature-first vs Layer-first, Monorepo vs отдельные репозитории)
- Автоматическая валидация выбора технологий
- Запись решений для каждого компонента стека в Decision Log
- Автоматическое обновление прогресса через скрипт

**Выходные артефакты**:
- `.pldf/tech-stack.md` - выбранный стек
- `.pldf/tech-rationale.md` - детальное обоснование
- `.pldf/project-context.md` - обновленный контекст проекта
- `.pldf/memory/progress.json` - обновленный файл прогресса

### Этап 3: Архитектура (`/pldf.architecture`)
Проектирование архитектуры, структуры проекта, модели данных и API контрактов на основе HTML прототипов.

**Выходные артефакты**:
- `.pldf/architecture.md` - архитектура проекта
- `.pldf/data-model.md` - модель данных
- `.pldf/contracts/` - API контракты (создаются на основе элементов из прототипов)

**Особенность**: API контракты создаются на основе анализа HTML прототипов - если в прототипе нет элемента, для него не создается эндпоинт.

### Этап 4: План реализации (`/pldf.plan`)
Разбивка проекта на конкретные шаги с зависимостями и тестами.

**Особенности**:
- **Инкрементальный подход**: каждый запуск команды создает только один новый шаг (работа в пределах контекстного окна)
- Автоматическое определение следующего шага на основе зависимостей и приоритетов (P1 → P2 → P3)
- Кэширование контекста планирования (`.pldf/planning-context-cache.md`) для оптимизации инкрементального режима
- Визуализация прогресса планирования после каждого шага (покрытие функций P1, P2, P3)
- Автоматическое отслеживание метрик покрытия функций по приоритетам
- Запись решений о подходе к разбивке и стратегии тестирования в Decision Log

**Выходные артефакты**:
- `.pldf/implementation-plan.md` - план реализации (обновляется инкрементально)
- `.pldf/steps/step-[NN]-[name]/` - директория с описаниями шагов (создается по одному за запуск)
- `.pldf/memory/progress.json` - файл отслеживания прогресса с метриками планирования
- `.pldf/planning-context-cache.md` - кэш контекста планирования (создается в initial mode)
- `.pldf/project-context.md` - обновленный контекст проекта

### Этап 5: Реализация (`/pldf.implement`)
Пошаговая реализация проекта с автоматической валидацией. HTML прототипы используются как референс для реализации.

**Особенности**:
- Последовательное выполнение шагов с учетом зависимостей
- Использование HTML прототипов из `.pldf/ui-prototype/` как визуального гайда при реализации интерфейса
- Автоматическая валидация каждого шага перед переходом к следующему (чек-лист из 7 пунктов)
- **Обязательные коммиты в GIT после каждого успешно завершенного шага** (только после валидации)
- Ручное обновление `.pldf/memory/progress.json` после каждого успешно завершенного шага
- Возможность возобновления с любого шага (определение начального шага из `.pldf/memory/progress.json`)

**Выходные артефакты**:
- Реализованный код проекта
- Обновленный `.pldf/memory/progress.json` (после каждого шага)
- История коммитов GIT с коммитом для каждого завершенного шага

### Этап 6: Review и рефлексия (`/pldf.review`)
Анализ созданных артефактов, выявление сильных и слабых сторон, сравнение с лучшими практиками и проведение рефлексии над процессом разработки.

**Выходные артефакты**:
- `.pldf/review.md` - детальный отчет с анализом и рекомендациями
- `.pldf/improvements.md` - приоритизированный список улучшений

**Особенность**: Этот этап помогает закрепить полученные знания, выявить области для улучшения и провести рефлексию над процессом разработки.

### Команда тестирования (`/pldf.test`)
Тестирование студента на основе артефактов текущего и всех предыдущих этапов. Команда может быть вызвана в любой момент для проверки понимания материала.

**Особенности**:
- Генерирует тесты трех типов: выбор одного варианта, выбор нескольких вариантов, свободный ответ
- Интерактивный режим опроса (вопросы задаются по очереди)
- Учитывает историю предыдущих попыток тестирования
- Адаптивная генерация вопросов: фокус на слабых местах, более сложные вопросы для развития
- Детальный отчет с разбивкой по этапам и темам
- Сохранение истории всех попыток с отслеживанием прогресса

**Выходные артефакты**:
- `.pldf/assessments.json` - файл с историей всех попыток тестирования и статистикой

**Особенность**: Команда может быть вызвана в любой момент после создания хотя бы одного артефакта проекта. Система автоматически определяет текущий этап и генерирует вопросы на основе всех существующих артефактов.

### Команда подсказок (`/hint` или `/pldf.hint`)
Получение интеллектуальных подсказок для помощи в процессе разработки.

**Особенности**:
- Подсказки даются в виде наводящих вопросов, а не прямых ответов
- Контекстные подсказки на основе текущего этапа и ошибок валидации
- Интеграция с библиотекой паттернов (автоматическое предложение релевантных паттернов)
- Предложение дополнительных ресурсов для изучения

**Использование**:
- `/hint` - подсказка для текущего этапа
- `/hint concept` - подсказка для этапа концепции
- `/hint design api` - подсказка по конкретной теме

## Быстрый старт

См. [`QUICKSTART.md`](QUICKSTART.md) для быстрого старта.

1. **Начните с концепции**:
   ```
   /pldf.concept Описание вашей идеи проекта
   ```
   Агент создаст концепцию, инициализирует GIT репозиторий с `.gitignore` и создаст первый коммит.

2. **Создайте дизайн UI с HTML прототипами**:
   ```
   /pldf.design
   ```
   После создания запустите локальный сервер (инструкция в `.pldf/ui-prototype/README.md`) и откройте `.pldf/ui-prototype/index.html` в браузере для просмотра прототипов. **Важно**: Утвердите дизайн перед продолжением - агент не перейдет к следующему этапу без вашего утверждения.

4. **Выберите технологии**:
   ```
   /pldf.tech
   ```

5. **Спроектируйте архитектуру**:
   ```
   /pldf.architecture
   ```

6. **Создайте план реализации** (инкрементально):
   ```
   /pldf.plan
   ```
   Команда создает один шаг за запуск. Запускайте команду повторно для планирования следующего шага. Система автоматически определит следующий шаг на основе зависимостей и приоритетов.

7. **Начните реализацию**:
   ```
   /pldf.implement
   ```

8. **Проведите review и рефлексию** (рекомендуется):
   ```
   /pldf.review
   ```

## Просмотр HTML прототипов

После создания прототипов на этапе `/pldf.design` агент создаст локальный сервер для просмотра прототипов.

### Запуск локального сервера

Инструкция по запуску находится в файле `.pldf/ui-prototype/README.md`. Агент создаст сервер на основе доступных технологий (Go, Python или Node.js).

### Просмотр в Cursor
1. Запустите локальный сервер согласно инструкции в `.pldf/ui-prototype/README.md`
2. Откройте файл `.pldf/ui-prototype/index.html`
3. Нажмите `Ctrl+Shift+P` (или `Cmd+Shift+P` на Mac)
4. Выберите команду `Simple Browser: Show`
5. Или просто кликните правой кнопкой на файл и выберите "Open with Live Server" (если установлен)

### Просмотр в обычном браузере
1. Запустите локальный сервер согласно инструкции в `.pldf/ui-prototype/README.md`
2. Откройте файл `.pldf/ui-prototype/index.html` в любом браузере

## Скрипты

### Валидация этапов

PLDF включает систему валидации для каждого этапа разработки. Валидация автоматически выполняется агентом при завершении каждого этапа, но вы также можете запустить ее вручную:

#### Валидация концепции
```powershell
.\.pldf\scripts\validate-concept.ps1
```


#### Валидация дизайна
```powershell
.\.pldf\scripts\validate-design.ps1
```

#### Валидация технологий
```powershell
.\.pldf\scripts\validate-tech.ps1
```

#### Валидация архитектуры
```powershell
.\.pldf\scripts\validate-architecture.ps1
```

#### Валидация плана
```powershell
.\.pldf\scripts\validate-plan.ps1
```

Все скрипты поддерживают флаг `-Json` для вывода результатов в формате JSON:
```powershell
.\.pldf\scripts\validate-concept.ps1 -Json
```

**Важно**: Агент автоматически выполняет валидацию перед переходом к следующему этапу. Если валидация не пройдена, агент не может продолжить работу до исправления ошибок.

### Валидация шага реализации
```powershell
.\.pldf\scripts\validate-step.ps1 -StepNumber "01"
```

### Запуск тестов
```powershell
.\.pldf\scripts\run-tests.ps1 -StepNumber "01"
```

### Настройка окружения
```powershell
.\.pldf\scripts\setup-environment.ps1
```

### Обновление прогресса
```powershell
.\.pldf\scripts\update-progress.ps1 -StepNumber "01" -Status "completed"
```

## Система подсказок и наставничества

PLDF включает интеллектуальную систему подсказок, которая помогает студентам при возникновении проблем.

### Контекстные подсказки

При ошибках валидации система автоматически предлагает подсказки:
- Подсказки появляются в выводе скриптов валидации
- Содержат наводящие вопросы, а не прямые ответы
- Предлагают дополнительные ресурсы для изучения

**Пример**: Если создан эндпоинт для элемента, которого нет в прототипе, система предложит: "Проверьте прототип еще раз. API контракты должны создаваться только на основе элементов, которые есть в UI прототипе."

### Подсказки по запросу

Используйте команду `/hint` для получения подсказок:
```
/hint                    # Подсказка для текущего этапа
/hint concept           # Подсказка для этапа концепции
/hint design api        # Подсказка по конкретной теме
```

Команда анализирует текущий контекст и предоставляет релевантные наводящие вопросы.

### Дополнительные ресурсы

Система подсказок включает базу дополнительных ресурсов:
- Ссылки на документацию технологий
- Статьи и туториалы по темам

### Структура подсказок

Подсказки хранятся в:
- `.pldf/hints/hints.json` - база подсказок по этапам и категориям ошибок
- `.pldf/hints/resources.json` - дополнительные ресурсы

## Образовательные элементы

Каждый этап включает:

- **Почему это важно?** - объяснение важности этапа
- **Частые ошибки студентов** - что может пойти не так и как избежать
- **Вопросы для размышления** - вопросы для закрепления материала
- **Проверка понимания** - мини-тесты и практические задания для оценки усвоения материала
  - Мини-тест из 3-5 вопросов по ключевым концепциям
  - Практическое задание для применения знаний
  - Адаптация сложности под уровень студента (начальный/продвинутый)
- **Подсказки и помощь** - раздел с типичными ошибками, подсказками и ссылками на ресурсы

**Дополнительная проверка понимания**: Используйте команду `/pldf.test` в любой момент для прохождения тестирования на основе всех созданных артефактов. Команда генерирует адаптивные тесты, учитывающие историю предыдущих попыток, и предоставляет детальный отчет с рекомендациями.

## Система метрик и аналитики

PLDF включает систему метрик для отслеживания прогресса разработки проекта.

### Доступные метрики

#### Метрики прогресса планирования

На этапе `/pldf.plan` автоматически отслеживаются метрики покрытия функций по приоритетам:

- **p1Coverage** - покрытие функций приоритета P1 (критичные функции)
- **p2Coverage** - покрытие функций приоритета P2 (важные функции)
- **p3Coverage** - покрытие функций приоритета P3 (желательные функции)
- **overallProgress** - общий прогресс как взвешенное среднее (P1: 50%, P2: 30%, P3: 20%)

Эти метрики автоматически обновляются агентом при создании каждого нового шага плана и сохраняются в `.pldf/memory/progress.json` в разделе `planningMetadata.progressMetrics`.

#### Отслеживание прогресса шагов

В файле `.pldf/memory/progress.json` отслеживается:

- **currentStep** - текущий шаг реализации
- **completedSteps** - список завершенных шагов
- **plannedSteps** - список запланированных шагов
- **stepStatus** - статус каждого шага (completed, in-progress, pending)

Прогресс автоматически обновляется агентом на этапах `/pldf.plan` и `/pldf.implement`.

#### Анализ качества кода

На этапе `/pldf.review` агент выполняет ручной анализ качества кода:

- Соответствие принципам SOLID
- Читаемость и поддерживаемость кода
- Обработка ошибок
- Тестовое покрытие
- Выявление антипаттернов и технического долга

Анализ выполняется агентом вручную на основе изучения кодовой базы и артефактов проекта, а не через автоматические скрипты.

### Просмотр метрик

Для просмотра метрик прогресса используйте:

```powershell
# Просмотр прогресса (PowerShell)
Get-Content .pldf\memory\progress.json | ConvertFrom-Json

# Просмотр прогресса (Bash/Linux/macOS)
cat .pldf/memory/progress.json | jq .
```

Файл `.pldf/memory/progress.json` содержит:
- Текущий шаг и статусы всех шагов
- Метрики покрытия функций (P1, P2, P3)
- Общий прогресс планирования

## Система трейсинга решений (Decision Log)

PLDF включает систему трейсинга решений для отслеживания истории всех важных решений, принятых в процессе разработки проекта. Это помогает студентам понимать связи между этапами и обоснования принятых решений.

### Где хранится Decision Log

Decision Log ведется в файле `.pldf/project-context.md` в разделе "Decision Log (История решений)".

### Что записывается

На каждом этапе (кроме реализации и review) агент автоматически записывает важные решения с:
- **Датой** принятия решения
- **Этапом**, на котором было принято решение
- **Причиной** - детальным обоснованием
- **Альтернативами** - рассмотренными вариантами и их плюсами/минусами
- **Влиянием на следующие этапы** - как решение повлияло на архитектуру, план и т.д.
- **Связанными решениями** - ссылками на другие решения, которые связаны с этим

### Примеры решений

- **Этап concept**: Решения о приоритизации функций (P1, P2, P3), исключениях из MVP, определении платформ
- **Этап design**: Выбор паттернов UI, структуры навигации, группировки функций, адаптация под платформы
- **Этап tech**: Выбор каждого компонента стека (язык, фреймворки, БД, инструменты), структура проекта (Monorepo vs отдельные репозитории)
- **Этап architecture**: Выбор архитектурных паттернов, структуры проекта, подхода к API
- **Этап plan**: Подход к инкрементальному планированию, разбивке на шаги, стратегия тестирования, кэширование контекста

### Преимущества

1. **Понимание связей** - видно, как решения на одном этапе влияют на следующие
2. **Обоснование выбора** - все решения имеют детальное обоснование
3. **Альтернативы** - видны рассмотренные варианты и причины отказа от них
4. **Образовательная ценность** - студент видит полную картину принятия решений

## Преимущества HTML прототипов

1. **Визуальная обратная связь** - студент видит реальный интерфейс, а не описание
2. **Легче утверждать** - можно открыть в браузере и сразу увидеть результат
3. **Референс для реализации** - прототипы используются как референс на этапе реализации
4. **Исключение лишних эндпоинтов** - API контракты создаются только на основе элементов из прототипов

## Интеграция с другими фреймворками

PLDF может использоваться вместе с другими фреймворками (например, [SpecKit](https://github.com/github/spec-kit/)):

- PLDF отвечает за **образовательный процесс** и **методичность**
- Другие фреймворки могут использоваться для **технической реализации**

### Библиотека паттернов

Библиотека паттернов содержит best practices для различных аспектов разработки. Паттерны интегрированы в систему подсказок и доступны через команду `/hint`:

- **[auth-patterns/](.pldf/hints/patterns/auth-patterns/)** - Паттерны аутентификации
  - JWT токены, сессионная аутентификация, OAuth 2.0

- **[clean-architecture/](.pldf/hints/patterns/clean-architecture/)** - Чистая архитектура и структура проектов
  - Clean Architecture, Hexagonal Architecture, Onion Architecture
  - Структура Flutter, React, Backend проектов

- **[data-fetching/](.pldf/hints/patterns/data-fetching/)** - Паттерны получения данных
  - REST API клиенты, GraphQL, WebSocket

- **[state-management/](.pldf/hints/patterns/state-management/)** - Управление состоянием
  - Flutter: Provider, Riverpod, BLoC, GetX
  - Compose Multiplatform: StateFlow, Compose State, ViewModel
  - React: Redux, Context API, Zustand, Jotai
  - Vue: Vuex, Pinia
  - Angular: NgRx, Services

- **[error-handling/](.pldf/hints/patterns/error-handling/)** - Обработка ошибок
  - Error Boundaries, обработка ошибок API, логирование

См. [.pldf/hints/patterns/README.md](.pldf/hints/patterns/README.md) для подробной информации.

**Интеграция с системой подсказок**: Паттерны автоматически предлагаются командой `/hint` на соответствующих этапах разработки. Например, при работе над архитектурой команда `/hint architecture` может предложить релевантные паттерны из библиотеки.

## Команды

- `/pldf.concept` - Создание концепции проекта (инициализация GIT, создание `.gitignore`, первый коммит)
- `/pldf.design` - Создание дизайна UI с HTML прототипами (требует утверждения перед продолжением)
- `/pldf.tech` - Выбор технологического стека (обсуждение каждого выбора со студентом)
- `/pldf.architecture` - Проектирование архитектуры (API контракты на основе HTML прототипов)
- `/pldf.plan` - Создание плана реализации (инкрементально, один шаг за запуск)
- `/pldf.implement` - Пошаговая реализация (обязательные коммиты после каждого шага)
- `/pldf.review` - Review и рефлексия (анализ артефактов, выявление улучшений, не обязателен)
- `/pldf.test` - Тестирование студентов (может быть вызвано в любой момент для проверки понимания материала)
- `/hint` или `/pldf.hint` - Получение подсказок по текущему этапу или конкретной проблеме
