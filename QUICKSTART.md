# Быстрый старт с PLDF (Progressive Learning Development Framework)

Это краткое руководство поможет вам начать работу с **PLDF (Progressive Learning Development Framework)** — образовательным фреймворком для обучения разработке программного обеспечения с помощью LLM-агентов.

**PLDF фокусируется на понимании "ПОЧЕМУ" принимаются решения, а не только на технической реализации.**

## Способы вызова команд

Все команды PLDF можно вызывать двумя способами:

1. **Без параметров** - агент задаст уточняющие вопросы и проведет диалог:
   ```
   /pldf.concept
   /pldf.design
   /pldf.tech
   ```

2. **С пожеланиями** - передайте свои требования сразу, агент учтет их при работе:
   ```
   /pldf.concept Я хочу создать приложение для управления задачами
   /pldf.tech Хочу использовать Python 3.13 для бэкенда и Flutter для клиента
   /pldf.implement Хочу использовать async/await для всех асинхронных операций
   ```

Выбирайте способ в зависимости от ситуации: если у вас есть четкое видение - передавайте параметры сразу, если нужна помощь в формулировании - вызывайте команду без параметров.

## Шаг 1: Создайте концепцию проекта

Запустите команду одним из способов:

**Способ 1: С описанием идеи** (команда сразу начнет работу с вашими пожеланиями):
```
/pldf.concept Я хочу создать приложение для управления задачами, где пользователи могут создавать, редактировать и отмечать задачи как выполненные
```

**Способ 2: Без параметров** (команда задаст уточняющие вопросы):
```
/pldf.concept
```

Агент создаст файл `.pldf/concept.md` с детальной концепцией проекта.

**Что происходит на этом этапе:**
- Определение проблемы и целевой аудитории
- Приоритизация функций разрабатываемой системы (P1, P2, P3)
- Определение критериев успеха
- **Автоматическая инициализация GIT репозитория** с детальным `.gitignore` (исключает секреты, зависимости, временные файлы)
- **Автоматическая валидация** концепции перед переходом к следующему этапу
- **Запись решений** в Decision Log (`.pldf/project-context.md`)
- **Первый коммит** в GIT после создания концепции

## Шаг 2: Создайте дизайн UI с HTML прототипами

Запустите команду одним из способов:

**Способ 1: Без параметров** (рекомендуется - агент создаст дизайн на основе концепции):
```
/pldf.design
```

**Способ 2: С пожеланиями по дизайну** (если у вас есть конкретные предпочтения):
```
/pldf.design Хочу использовать минималистичный дизайн с темной темой, bottom navigation для мобильной версии
```

Агент создаст интерактивные HTML/CSS прототипы интерфейса на основе концепции и выбранных платформ. Дизайн будет адаптирован под выбранные платформы (мобильные паттерны для Mobile, десктопные для Desktop). 

**Что происходит на этом этапе:**
- Создание HTML/CSS прототипов для всех экранов MVP
- Адаптация дизайна под платформы из концепции (мобильные/десктопные паттерны)
- Определение пользовательских потоков
- Определение данных для каждого экрана
- **Создание локального сервера** для просмотра прототипов (Go, Python или Node.js)
- **Автоматическая валидация** дизайна (покрытие функций, логика потоков, соответствие платформам)
- **Запись решений** о паттернах UI и структуре навигации в Decision Log
- **Требование утверждения** дизайна студентом перед продолжением - агент не перейдет к следующему этапу без утверждения

**Важно**: 
1. Запустите локальный сервер (инструкция в `.pldf/ui-prototype/README.md`)
2. Откройте файл `.pldf/ui-prototype/index.html` в браузере
3. Просмотрите все экраны прототипа
4. Утвердите дизайн перед продолжением (ответьте "да" на вопрос агента)

**Как открыть прототип в Cursor**:
- Запустите локальный сервер согласно инструкции в `.pldf/ui-prototype/README.md`
- Откройте файл `.pldf/ui-prototype/index.html`
- Нажмите `Ctrl+Shift+P` (или `Cmd+Shift+P` на Mac)
- Выберите `Simple Browser: Show`
- Или просто откройте файл в любом браузере

**Почему это важно:** HTML прототипы используются как референс на этапе реализации и определяют, какие API контракты действительно нужны (исключая создание ненужных эндпоинтов).

## Шаг 3: Выберите технологии

Запустите команду одним из способов:

**Способ 1: Без параметров** (агент предложит варианты и обсудит их с вами):
```
/pldf.tech
```

**Способ 2: С требованиями к технологиям** (если у вас есть конкретные предпочтения или ограничения):
```
/pldf.tech Хочу использовать Python 3.13 для бэкенда и Flutter 3.38 для мобильного приложения
```

Агент предложит варианты технологий с обоснованием или учтет ваши требования. Обсудите варианты и выберите подходящий стек.

**Что происходит на этом этапе:**
- Анализ требований проекта на основе концепции и дизайна
- Предложение 2-3 вариантов для каждого компонента стека (язык, фреймворк, БД, инструменты)
- Детальное обоснование каждого варианта (плюсы, минусы, когда использовать)
- **Обсуждение каждого выбора со студентом** (не переход к следующему этапу без согласия)
- **Консультация по структуре проекта** (Feature-first vs Layer-first, Monorepo vs отдельные репозитории)
- **Автоматическая валидация** выбора технологий
- **Запись решений** для каждого компонента стека в Decision Log с обоснованием
- **Автоматическое обновление прогресса** через скрипт

**Образовательная ценность:** На этом этапе вы понимаете "ПОЧЕМУ" выбирается та или иная технология, а не просто следуете рекомендациям.

## Шаг 4: Спроектируйте архитектуру

Запустите команду одним из способов:

**Способ 1: Без параметров** (рекомендуется - агент создаст архитектуру на основе всех предыдущих этапов):
```
/pldf.architecture
```

**Способ 2: С предпочтениями по архитектуре** (если у вас есть конкретные требования):
```
/pldf.architecture Хочу использовать Clean Architecture с разделением на слои domain, data и presentation
```

Агент создаст детальную архитектуру проекта, модель данных и API контракты.

**Что происходит на этом этапе:**
- Проектирование структуры директорий проекта
- Создание модели данных на основе UI дизайна
- **Создание API контрактов на основе HTML прототипов** (только для элементов, которые есть в прототипе)
- Определение внешних зависимостей и интеграций
- **Автоматическая валидация** архитектуры
- **Запись решений** об архитектурных паттернах в Decision Log

**Важно:** API контракты создаются только на основе элементов из HTML прототипов. Если в прототипе нет элемента, для него не создается эндпоинт.

## Шаг 5: Создайте план реализации (инкрементально)

Запустите команду одним из способов:

**Способ 1: Без параметров** (рекомендуется - агент создаст следующий шаг автоматически):
```
/pldf.plan
```

**Способ 2: С предпочтениями по планированию** (если у вас есть идеи о следующем шаге):
```
/pldf.plan Следующим шагом хочу реализовать систему аутентификации пользователей
```

**Важно**: Команда использует инкрементальный подход - каждый запуск создает только один новый шаг. Запускайте команду повторно для планирования следующего шага.

**Что происходит на этом этапе:**
- **Инкрементальное планирование**: один шаг за запуск (работа в пределах контекстного окна)
- Автоматическое определение следующего шага на основе зависимостей и приоритетов (P1 → P2 → P3)
- Разбивка проекта на логические шаги
- Определение зависимостей между шагами
- Создание описаний шагов с задачами и тестами
- **Кэширование контекста планирования** (`.pldf/planning-context-cache.md`) для оптимизации
- **Визуализация прогресса планирования** после каждого шага (покрытие функций P1, P2, P3)
- Создание/обновление файла `.pldf/memory/progress.json` с метриками планирования
- **Автоматическая валидация** каждого шага
- **Запись решений** о подходе к разбивке и стратегии тестирования в Decision Log

## Шаг 6: Начните реализацию

Запустите команду одним из способов:

**Способ 1: Без параметров** (агент начнет с текущего шага из `.pldf/memory/progress.json`):
```
/pldf.implement
```

**Способ 2: С указанием конкретного шага** (если хотите начать или возобновить с определенного шага):
```
/pldf.implement step-03-auth
```

**Способ 3: С дополнительными пожеланиями** (если у вас есть конкретные требования к реализации):
```
/pldf.implement Хочу использовать async/await для всех асинхронных операций
```

Агент будет выполнять шаги последовательно, валидировать каждый шаг и обновлять прогресс.

**Что происходит на этом этапе:**
- Последовательное выполнение шагов с учетом зависимостей
- **Использование HTML прототипов из `.pldf/ui-prototype/` как визуального гайда** при реализации интерфейса
- Автоматическая валидация каждого шага перед переходом к следующему (чек-лист из 7 пунктов)
- **Обязательные коммиты в GIT** после каждого успешно завершенного шага (только после валидации)
- Ручное обновление прогресса в `.pldf/memory/progress.json` после каждого успешно завершенного шага
- Возможность возобновления с любого шага (определение начального шага из `.pldf/memory/progress.json`)

**Важно:**
- Каждый шаг валидируется перед переходом к следующему (нельзя перейти без валидации)
- Коммиты создаются только после успешной валидации шага
- HTML прототипы используются как визуальный гайд для реализации интерфейса
- Прогресс обновляется вручную после каждого успешно завершенного шага

## Шаг 7: Проведите review и рефлексию (рекомендуется)

Запустите команду одним из способов:

**Способ 1: Без параметров** (агент проанализирует все артефакты автоматически):
```
/pldf.review
```

**Способ 2: С фокусом на конкретные аспекты** (если хотите углубить анализ определенных частей):
```
/pldf.review Особое внимание на качество кода и соответствие SOLID принципам
```

Агент проанализирует созданные артефакты, выявит сильные и слабые стороны, сравнит с лучшими практиками и проведет рефлексию над процессом разработки. Это поможет закрепить полученные знания и улучшить качество кода.

**Что происходит на этом этапе:**
- Анализ концепции и соответствия реализации
- **Анализ качества кода агентом** (SOLID, читаемость, обработка ошибок, тестовое покрытие) - выполняется вручную агентом на основе изучения кодовой базы
- Сравнение реализации с HTML-прототипами
- Анализ технологических решений
- Сравнение с лучшими практиками индустрии
- Выявление конкретных улучшений с приоритизацией
- Рефлексия над процессом разработки
- Проверка понимания для закрепления знаний

**Важно**: 
- Review не является обязательным, но крайне рекомендуется для закрепления знаний
- На этом этапе вы получите конкретные рекомендации по улучшению
- Анализ качества кода агентом выявит области для улучшения
- Рефлексия помогает понять, что было изучено и что требует повторения

## Системы фреймворка

PLDF включает несколько интегрированных систем, которые помогают в обучении и разработке:

### Система валидации

На каждом этапе выполняется **автоматическая валидация** перед переходом к следующему:
- Проверка соответствия критериям этапа
- Выявление ошибок и несоответствий
- Предложение подсказок при ошибках
- **Нельзя перейти к следующему этапу**, пока валидация не пройдена

Валидация выполняется автоматически агентом, но вы также можете запустить ее вручную (см. раздел "Полезные команды").

### Система метрик и аналитики

PLDF автоматически отслеживает:

- **Метрики прогресса планирования** - покрытие функций по приоритетам (P1, P2, P3) и общий прогресс
  - Автоматически обновляются на этапе `/pldf.plan` при создании каждого шага
  - Сохраняются в `.pldf/memory/progress.json` в разделе `planningMetadata.progressMetrics`

- **Отслеживание прогресса шагов** - текущий шаг, завершенные и запланированные шаги, статусы
  - Автоматически обновляется на этапах `/pldf.plan` и `/pldf.implement`
  - Сохраняется в `.pldf/memory/progress.json`

- **Анализ качества кода** - выполняется агентом вручную на этапе `/pldf.review`
  - Анализирует: SOLID, читаемость, обработку ошибок, тестовое покрытие
  - Выполняется на основе изучения кодовой базы и артефактов проекта

Для просмотра метрик используйте команду просмотра `.pldf/memory/progress.json` (см. раздел "Полезные команды").

### Система подсказок

При возникновении проблем используйте команду `/pldf.hint` одним из способов:

**Способ 1: Общая подсказка для текущего этапа**:
```
/pldf.hint
```

**Способ 2: Подсказка для конкретного этапа**:
```
/pldf.hint concept           # Подсказка для этапа концепции
/pldf.hint architecture      # Подсказка для этапа архитектуры
```

**Способ 3: Подсказка по конкретной теме**:
```
/pldf.hint design api        # Подсказка по API в дизайне
/pldf.hint concept functions # Подсказка по функциям в концепции
```

**Особенности:**
- Подсказки даются в виде **наводящих вопросов**, а не прямых ответов
- Контекстные подсказки автоматически предлагаются при ошибках валидации
- Интеграция с **библиотекой паттернов** - подсказки могут ссылаться на паттерны проектирования
- Предложение дополнительных ресурсов для изучения

### Decision Log (История решений)

Все важные решения отслеживаются в `.pldf/project-context.md` в разделе "Decision Log":
- Решения о приоритизации функций, выборе платформ, технологий, архитектурных паттернах
- Для каждого решения: дата, этап, причина, альтернативы, влияние на следующие этапы
- Помогает понять связи между этапами и обоснования принятых решений

Decision Log автоматически обновляется на каждом этапе (кроме реализации и review).

### Библиотека паттернов

PLDF включает обширную библиотеку паттернов проектирования:
- Паттерны аутентификации (JWT, OAuth, сессионная)
- Архитектурные паттерны (Clean Architecture, Hexagonal, Onion)
- Паттерны работы с данными (REST API, GraphQL, WebSocket)
- Управление состоянием (для Flutter, React, Vue, Angular, Compose Multiplatform)
- Обработка ошибок

Паттерны доступны через систему подсказок (`/pldf.hint`) и содержат детальные описания, примеры кода и рекомендации.

## Проверка понимания

На каждом этапе PLDF агент задает вопросы для проверки понимания:
- **Мини-тест** из 3-5 вопросов по ключевым концепциям
- **Практическое задание** для применения знаний
- **Адаптация сложности** под ваш уровень (начальный/продвинутый)

Это помогает убедиться, что вы понимаете материал перед переходом к следующему этапу.

### Дополнительное тестирование

Вы также можете использовать команду `/pldf.test` в любой момент для прохождения тестирования на основе всех созданных артефактов:

**Способ 1: Без параметров** (агент автоматически определит этап и проведет тестирование):
```
/pldf.test
```

**Способ 2: С указанием области** (если хотите сфокусироваться на конкретной теме, хотя команда пока не поддерживает этот параметр явно):
```
/pldf.test
```

**Особенности команды тестирования:**
- Генерирует тесты на основе текущего и всех предыдущих этапов
- Учитывает историю предыдущих попыток (избегает повторений, фокусируется на слабых местах)
- Предоставляет детальный отчет с разбивкой по этапам и темам
- Отслеживает прогресс и показывает улучшения

**Когда использовать:**
- После завершения любого этапа для проверки понимания
- Перед переходом к следующему этапу для закрепления знаний
- В любой момент для оценки текущего уровня понимания

## Советы

1. **Не пропускайте этапы** - каждый этап важен и строится на предыдущем. Валидация не позволит перейти к следующему этапу, пока текущий не завершен правильно.

2. **Утверждайте дизайн** - просмотрите HTML прототипы в браузере и утвердите дизайн перед продолжением. Это сэкономит время на переделку кода.

3. **Обсуждайте технологии** - не просто соглашайтесь с предложениями, задавайте вопросы. Понимание выбора развивает инженерное мышление.

4. **Используйте систему подсказок** - если что-то непонятно, используйте `/pldf.hint`. Подсказки даются в виде наводящих вопросов, что помогает учиться самостоятельно.

5. **Просматривайте Decision Log** - история решений в `.pldf/project-context.md` помогает понять обоснования и связи между этапами. Это особенно полезно при возвращении к проекту после перерыва.

6. **Просматривайте метрики прогресса** - периодически просматривайте `.pldf/memory/progress.json` для отслеживания прогресса. Метрики показывают покрытие функций по приоритетам и общий прогресс планирования.

7. **Отвечайте на вопросы проверки понимания** - это помогает закрепить знания. Не пропускайте мини-тесты и практические задания.

8. **Проводите review** - анализ проекта помогает выявить области для улучшения и закрепить полученные знания. Анализ качества кода агентом покажет конкретные рекомендации.
