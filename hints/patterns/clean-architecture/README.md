## Общие принципы чистой архитектуры

1. **Независимость от фреймворков** - бизнес-логика не зависит от UI, БД, внешних библиотек
2. **Тестируемость** - бизнес-логика может быть протестирована без UI, БД, веб-сервера
3. **Независимость от UI** - UI можно легко заменить без изменения бизнес-логики
4. **Независимость от БД** - можно заменить Oracle на SQL Server, MongoDB и т.д.
5. **Независимость от внешних сервисов** - бизнес-логика не знает о внешнем мире

## Правило зависимостей

**Зависимости направлены внутрь**: внешние слои зависят от внутренних, но не наоборот.

```
UI → Use Cases → Entities
     ↓
  Frameworks (DB, Web, etc.)
```

## Выбор архитектуры

Выбор зависит от:
- **Размера проекта** - простые проекты могут обойтись простой структурой
- **Сложности бизнес-логики** - сложная логика требует более строгого разделения
- **Команды** - опыт команды влияет на выбор
- **Требований к тестируемости** - чистая архитектура упрощает тестирование
- **Долгосрочности проекта** - для долгосрочных проектов важна гибкость

## Когда использовать

✅ **Используйте чистую архитектуру когда:**
- Проект будет долгосрочным
- Бизнес-логика сложная
- Нужна высокая тестируемость
- Команда большая и нужна четкая структура
- Возможны изменения в технологиях (БД, UI фреймворк)

❌ **Не используйте когда:**
- Проект очень простой (MVP, прототип)
- Команда маленькая и нет времени на структуру
- Проект краткосрочный
- Нет сложной бизнес-логики
